# 보안 정책

OpenClaw에서 보안 문제를 발견했다고 판단되면 비공개로 신고해 주세요.

## 신고

문제가 있는 저장소에 직접 취약점을 신고하세요:

- **Core CLI 및 게이트웨이** — [openclaw/openclaw](https://github.com/openclaw/openclaw)
- **macOS 데스크톱 앱** — [openclaw/openclaw](https://github.com/openclaw/openclaw) (apps/macos)
- **iOS 앱** — [openclaw/openclaw](https://github.com/openclaw/openclaw) (apps/ios)
- **Android 앱** — [openclaw/openclaw](https://github.com/openclaw/openclaw) (apps/android)
- **ClawHub** — [openclaw/clawhub](https://github.com/openclaw/clawhub)
- **신뢰 및 위협 모델** — [openclaw/trust](https://github.com/openclaw/trust)

특정 저장소에 해당하지 않거나 확실하지 않은 경우 **[security@openclaw.ai](mailto:security@openclaw.ai)** 로 이메일을 보내주시면 적절한 곳으로 전달하겠습니다.

전체 신고 지침은 [Trust 페이지](https://trust.openclaw.ai)를 참조하세요.

### 신고 시 필수 항목

1. **제목**
2. **심각도 평가**
3. **영향**
4. **영향받는 구성요소**
5. **기술적 재현 방법**
6. **입증된 영향**
7. **환경**
8. **수정 제안**

재현 단계, 입증된 영향, 수정 제안이 없는 신고는 우선순위가 낮아집니다. AI 생성 스캐너 결과의 양이 많기 때문에 문제를 이해하는 연구자의 검증된 신고를 받아야 합니다.

### 신고 수락 기준 (빠른 분류 경로)

가장 빠른 분류를 위해 다음을 모두 포함하세요:

- 현재 리비전의 정확한 취약 경로 (`file`, 함수, 줄 범위).
- 테스트된 버전 세부 정보 (OpenClaw 버전 및/또는 커밋 SHA).
- 최신 `main` 또는 최신 릴리스 버전에 대한 재현 가능한 PoC.
- OpenClaw의 문서화된 신뢰 경계에 연결된 입증된 영향.
- 노출된 비밀 신고의 경우: 해당 자격 증명이 OpenClaw 소유임(또는 OpenClaw 운영 인프라/서비스에 대한 접근 권한을 부여함)을 증명.
- 적대적 운영자가 하나의 게이트웨이 호스트/설정을 공유하는 것에 의존하지 않는다는 명시적 선언.
- 아래 범위 외 섹션에 해당하지 **않는** 이유를 설명하는 범위 확인.

이러한 요구 사항을 충족하지 못하는 신고는 `invalid` 또는 `no-action`으로 종료될 수 있습니다.

### 일반적인 오탐 패턴

다음은 자주 신고되지만 일반적으로 코드 변경 없이 종료됩니다:

- 경계 우회 없는 프롬프트 주입 전용 체인 (프롬프트 주입은 범위 외).
- 원격 주입으로 제시된 운영자 의도 로컬 기능 (예: TUI 로컬 `!` 셸).
- 권한 상승으로 제시된 인가된 사용자 트리거 로컬 작업. 예시: 허용 목록/소유자 발신자가 `/export-session /absolute/path.html`을 실행하여 호스트에 쓰기. 이 신뢰 모델에서 인가된 사용자 작업은 인증/샌드박스/경계 우회를 입증하지 않는 한 신뢰된 호스트 작업입니다.
- 공유 게이트웨이 호스트/설정에서 사용자별 멀티테넌트 인가를 가정하는 신고.
- 신뢰 경계 우회 없이 신뢰된 운영자 설정 입력(예: `sessionFilter` 또는 `logging.redactPatterns`의 재난적 정규식)을 필요로 하는 ReDoS/DoS 주장.
- 기본 로컬/루프백 배포에서의 HSTS 누락 발견.
- HTTP 모드가 이미 서명 비밀 검증을 사용하는 경우의 Slack 웹훅 서명 발견.
- 이 저장소의 Discord 통합에서 사용하지 않는 경로에 대한 Discord 인바운드 웹훅 서명 발견.
- 오래된/존재하지 않는 경로에 대한 스캐너 전용 주장, 또는 작동하는 재현 없는 주장.

## 운영자 신뢰 모델 (중요)

OpenClaw는 하나의 게이트웨이를 멀티테넌트, 적대적 사용자 경계로 모델링하지 **않습니다**.

- 인증된 게이트웨이 호출자는 해당 게이트웨이 인스턴스의 신뢰된 운영자로 취급됩니다.
- 세션 식별자(`sessionKey`, 세션 ID, 레이블)는 라우팅 컨트롤이며 사용자별 인가 경계가 아닙니다.
- 한 운영자가 동일 게이트웨이의 다른 운영자 데이터를 볼 수 있는 것은 이 신뢰 모델에서 예상되는 동작입니다.
- OpenClaw는 기술적으로 한 머신에서 여러 게이트웨이 인스턴스를 실행할 수 있지만 권장되는 운영 방식은 신뢰 경계별로 깨끗하게 분리하는 것입니다.
- 권장 모드: 머신(또는 VPS)당 한 사용자, 해당 사용자를 위한 하나의 게이트웨이, 그리고 해당 게이트웨이 내 하나 이상의 에이전트.
- 여러 사용자가 OpenClaw을 필요로 하는 경우 사용자당 하나의 VPS(또는 호스트/OS 사용자 경계)를 사용하세요.
- 고급 설정의 경우 한 머신에서 여러 게이트웨이가 가능하지만 엄격한 격리가 필요하며 기본 권장 사항은 아닙니다.
- 실행 동작은 기본적으로 호스트 우선입니다: `agents.defaults.sandbox.mode`의 기본값은 `off`입니다.
- `tools.exec.host`는 라우팅 선호도로 `sandbox`가 기본값이지만 세션에 샌드박스 런타임이 활성화되어 있지 않으면 게이트웨이 호스트에서 실행됩니다.
- 암시적 실행 호출(도구 호출에 명시적 호스트 없음)도 동일한 동작을 따릅니다.
- 이것은 OpenClaw의 단일 사용자 신뢰 운영자 모델에서 예상되는 동작입니다. 격리가 필요한 경우 샌드박스 모드(`non-main`/`all`)를 활성화하고 엄격한 도구 정책을 유지하세요.

## 범위 외

- 공용 인터넷 노출
- 문서에서 권장하지 않는 방식으로 OpenClaw 사용
- 상호 신뢰하지 않는/적대적 운영자가 하나의 게이트웨이 호스트와 설정을 공유하는 배포 (예: `sessions.list`, `sessions.preview`, `chat.history` 또는 유사한 컨트롤 플레인 읽기에 대한 운영자별 격리를 기대하는 신고)
- 프롬프트 주입 전용 공격 (정책/인증/샌드박스 경계 우회 없이)
- 신뢰된 로컬 상태(`~/.openclaw`, `MEMORY.md` / `memory/*.md` 같은 워크스페이스 파일)에 대한 쓰기 접근이 필요한 신고
- 인증, 샌드박스, 또는 기타 문서화된 경계를 우회하지 않고 이미 인가된 발신자가 의도적으로 로컬 작업 명령(예: 절대 호스트 경로에 쓰는 `/export-session`)을 호출하는 것만이 입증된 영향인 신고
- 운영자가 활성화한 `dangerous*`/`dangerously*` 설정 옵션이 기본값을 약화시킨다는 것만이 주장인 신고 (이것들은 설계상 명시적인 비상 트레이드오프)
- 가용성 영향을 트리거하기 위해 신뢰된 운영자 제공 설정 값(예: 사용자 정의 정규식 패턴)에 의존하는 신고. 이들은 여전히 심층 방어 강화로 수정될 수 있지만 보안 경계 우회는 아닙니다.
- OpenClaw 영향 입증 없이 타사/사용자 제어 자격 증명(OpenClaw 소유가 아니며 OpenClaw 운영 인프라/서비스에 대한 접근 권한을 부여하지 않는)인 노출된 비밀
- 경계 우회 없이 샌드박스 런타임이 비활성화/사용 불가능할 때 호스트 측 실행만이 주장인 신고 (신뢰 운영자 모델의 문서화된 기본 동작).

## 배포 가정

OpenClaw 보안 지침은 다음을 가정합니다:

- OpenClaw이 실행되는 호스트는 신뢰된 OS/관리자 경계 내에 있습니다.
- `~/.openclaw` 상태/설정(`openclaw.json` 포함)을 수정할 수 있는 사람은 사실상 신뢰된 운영자입니다.
- 상호 신뢰하지 않는 사람들이 공유하는 단일 게이트웨이는 **권장되지 않는 설정**입니다. 신뢰 경계별로 별도의 게이트웨이(또는 최소한 별도의 OS 사용자/호스트)를 사용하세요.
- 인증된 게이트웨이 호출자는 신뢰된 운영자로 취급됩니다. 세션 식별자(예: `sessionKey`)는 라우팅 컨트롤이며 사용자별 인가 경계가 아닙니다.
- 한 머신에서 여러 게이트웨이 인스턴스를 실행할 수 있지만 권장 모델은 사용자별 깨끗한 격리(사용자당 하나의 호스트/VPS 선호)입니다.

## 단일 사용자 신뢰 모델 (개인 어시스턴트)

OpenClaw의 보안 모델은 "개인 어시스턴트"(한 명의 신뢰된 운영자, 잠재적으로 여러 에이전트)이며 "공유 멀티테넌트 버스"가 아닙니다.

- 여러 사람이 동일한 도구 활성화 에이전트에 메시지를 보낼 수 있는 경우(예: 공유 Slack 워크스페이스), 모두 해당 에이전트에 부여된 권한 내에서 에이전트를 조종할 수 있습니다.
- 세션 또는 메모리 범위 지정은 컨텍스트 누출을 줄이지만 사용자별 호스트 인가 경계를 생성하지는 **않습니다**.
- 혼합 신뢰 또는 적대적 사용자의 경우 OS 사용자/호스트/게이트웨이로 격리하고 경계별로 별도의 자격 증명을 사용하세요.
- 사용자가 동일한 신뢰 경계에 있고 에이전트가 엄격하게 업무 전용인 경우 회사 공유 에이전트는 유효한 설정이 될 수 있습니다.
- 회사 공유 설정의 경우 전용 머신/VM/컨테이너와 전용 계정을 사용하세요. 해당 런타임에 개인 데이터를 혼합하지 마세요.
- 해당 호스트/브라우저 프로필이 개인 계정(예: Apple/Google/개인 비밀번호 관리자)에 로그인되어 있으면 경계가 붕괴되어 개인 데이터 노출 위험이 증가합니다.

## 에이전트 및 모델 가정

- 모델/에이전트는 신뢰된 주체가 **아닙니다**. 프롬프트/콘텐츠 주입이 동작을 조작할 수 있다고 가정하세요.
- 보안 경계는 호스트/설정 신뢰, 인증, 도구 정책, 샌드박싱, 실행 승인에서 비롯됩니다.
- 프롬프트 주입 자체는 이러한 경계 중 하나를 넘지 않는 한 취약점 신고가 아닙니다.

## 게이트웨이 및 노드 신뢰 개념

OpenClaw는 라우팅과 실행을 분리하지만 둘 다 동일한 운영자 신뢰 경계 내에 있습니다:

- **게이트웨이**는 컨트롤 플레인입니다. 호출자가 게이트웨이 인증을 통과하면 해당 게이트웨이의 신뢰된 운영자로 취급됩니다.
- **노드**는 게이트웨이의 실행 확장입니다. 노드를 페어링하면 해당 노드에 대한 운영자 수준 원격 기능이 부여됩니다.
- **실행 승인**(허용 목록/확인 UI)은 우발적 명령 실행을 줄이기 위한 운영자 가드레일이며 멀티테넌트 인가 경계가 아닙니다.
- 신뢰하지 않는 사용자 격리를 위해서는 신뢰 경계별로 분리하세요: 경계별로 별도의 게이트웨이와 별도의 OS 사용자/호스트.

## 워크스페이스 메모리 신뢰 경계

`MEMORY.md` 및 `memory/*.md`는 일반 워크스페이스 파일이며 신뢰된 로컬 운영자 상태로 취급됩니다.

- 워크스페이스 메모리 파일을 편집할 수 있다면 이미 신뢰된 운영자 경계를 넘은 것입니다.
- 해당 파일에 대한 메모리 검색 인덱싱/리콜은 예상되는 동작이며 샌드박스/보안 경계가 아닙니다.
- 범위 외로 간주되는 신고 패턴 예시: "공격자가 `memory/*.md`에 악성 콘텐츠를 작성한 후 `memory_search`가 이를 반환합니다."
- 상호 신뢰하지 않는 사용자 간 격리가 필요한 경우 OS 사용자 또는 호스트별로 분리하고 별도의 게이트웨이를 실행하세요.

## 플러그인 신뢰 경계

플러그인/확장은 게이트웨이와 **인프로세스**로 로드되며 신뢰된 코드로 취급됩니다.

- 플러그인은 OpenClaw 프로세스와 동일한 OS 권한으로 실행할 수 있습니다.
- 런타임 헬퍼(예: `runtime.system.runCommandWithTimeout`)는 편의 API이며 샌드박스 경계가 아닙니다.
- 신뢰할 수 있는 플러그인만 설치하고 명시적으로 신뢰된 플러그인 ID를 고정하기 위해 `plugins.allow`를 사용하는 것이 좋습니다.

## 운영 지침

위협 모델 + 보안 강화 지침(`openclaw security audit --deep` 및 `--fix` 포함)은 다음을 참조하세요:

- `https://docs.openclaw.ai/gateway/security`

### 도구 파일시스템 강화

- `tools.exec.applyPatch.workspaceOnly: true` (권장): `apply_patch` 쓰기/삭제를 설정된 워크스페이스 디렉토리 내로 제한합니다.
- `tools.fs.workspaceOnly: true` (선택): `read`/`write`/`edit`/`apply_patch` 경로를 워크스페이스 디렉토리로 제한합니다.
- 도구 실행을 트리거할 수 있는 사람을 완전히 신뢰하지 않는 한 `tools.exec.applyPatch.workspaceOnly: false`를 설정하지 마세요.

### 웹 인터페이스 안전성

OpenClaw의 웹 인터페이스(게이트웨이 Control UI + HTTP 엔드포인트)는 **로컬 전용**을 목적으로 합니다.

- 권장: 게이트웨이를 **루프백 전용**(`127.0.0.1` / `::1`)으로 유지하세요.
  - 설정: `gateway.bind="loopback"` (기본값).
  - CLI: `openclaw gateway run --bind loopback`.
- `gateway.controlUi.dangerouslyDisableDeviceAuth`는 로컬호스트 전용 비상 사용을 위한 것입니다.
  - OpenClaw는 설계상 배포 유연성을 유지하며 비로컬 설정을 하드 금지하지 않습니다.
  - 비로컬 및 기타 위험한 설정은 `openclaw security audit`에서 위험 소견으로 표시됩니다.
  - 이 운영자 선택 트레이드오프는 설계상이며 그 자체로는 보안 취약점이 아닙니다.
- Canvas 호스트 참고: 네트워크에 보이는 canvas는 신뢰된 노드 시나리오(LAN/tailnet)를 위해 **의도된 것**입니다.
  - 예상 설정: 비루프백 바인드 + 게이트웨이 인증(토큰/비밀번호/신뢰 프록시) + 방화벽/tailnet 제어.
  - 예상 경로: `/__openclaw__/canvas/`, `/__openclaw__/a2ui/`.
  - 이 배포 모델 자체만으로는 보안 취약점이 아닙니다.
- 공용 인터넷에 노출하지 **마세요** (`0.0.0.0`에 직접 바인드 금지, 공용 리버스 프록시 금지). 공용 노출에 대해 강화되어 있지 않습니다.
- 원격 접근이 필요한 경우 SSH 터널 또는 Tailscale serve/funnel을 사용하세요 (게이트웨이가 여전히 루프백에 바인드되도록), 그리고 강력한 게이트웨이 인증을 사용하세요.
- 게이트웨이 HTTP 표면에는 canvas 호스트(`/__openclaw__/canvas/`, `/__openclaw__/a2ui/`)가 포함됩니다. Canvas 콘텐츠를 민감/신뢰할 수 없는 것으로 취급하고 위험을 이해하지 않는 한 루프백 너머로 노출하지 마세요.

## 런타임 요구 사항

### Node.js 버전

OpenClaw은 **Node.js 22.12.0 이상** (LTS)이 필요합니다. 이 버전에는 중요한 보안 패치가 포함되어 있습니다:

- CVE-2025-59466: async_hooks DoS 취약점
- CVE-2026-21636: Permission model 우회 취약점

Node.js 버전을 확인하세요:

```bash
node --version  # v22.12.0 이상이어야 합니다
```

### Docker 보안

Docker에서 OpenClaw을 실행할 때:

1. 공식 이미지는 공격 표면을 줄이기 위해 비루트 사용자(`node`)로 실행됩니다
2. 추가 파일시스템 보호를 위해 가능한 경우 `--read-only` 플래그를 사용하세요
3. `--cap-drop=ALL`로 컨테이너 기능을 제한하세요

안전한 Docker 실행 예시:

```bash
docker run --read-only --cap-drop=ALL \
  -v openclaw-data:/app/data \
  openclaw/openclaw:latest
```
