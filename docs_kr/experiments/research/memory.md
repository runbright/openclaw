# 워크스페이스 메모리 v2 (오프라인): 리서치 노트

대상: Clawd 스타일 워크스페이스 (`agents.defaults.workspace`, 기본값 `~/.openclaw/workspace`) - "메모리"가 일일 하나의 Markdown 파일 (`memory/YYYY-MM-DD.md`)과 소수의 안정적인 파일 (예: `memory.md`, `SOUL.md`)로 저장됩니다.

이 문서는 Markdown을 정식의 검토 가능한 진실의 원천으로 유지하면서 파생 인덱스를 통해 **구조화된 회상** (검색, 엔티티 요약, 신뢰도 업데이트)을 추가하는 **오프라인 우선** 메모리 아키텍처를 제안합니다.

## 왜 변경하는가?

현재 설정 (하루에 하나의 파일)은 다음에 탁월합니다:

- "추가 전용" 저널링
- 사람이 편집
- git 기반 내구성 + 감사 가능성
- 저마찰 캡처 ("그냥 적어두기")

다음에 약합니다:

- 높은 회수율 검색 ("X에 대해 무엇을 결정했는가?", "Y를 마지막으로 시도한 때는?")
- 엔티티 중심 답변 ("Alice / The Castle / warelay에 대해 알려줘") - 많은 파일을 다시 읽지 않고
- 의견/선호도 안정성 (그리고 변경 시 증거)
- 시간 제약 ("2025년 11월 동안 사실이었던 것?") 및 충돌 해결

## 설계 목표

- **오프라인**: 네트워크 없이 작동; 노트북/Castle에서 실행 가능; 클라우드 의존성 없음.
- **설명 가능**: 검색된 항목은 귀속 가능(파일 + 위치)하고 추론과 분리 가능해야 함.
- **저의식**: 일일 로깅은 Markdown으로 유지, 무거운 스키마 작업 없음.
- **점진적**: v1은 FTS만으로 유용; 시맨틱/벡터 및 그래프는 선택적 업그레이드.
- **에이전트 친화적**: "토큰 예산 내 회상"을 쉽게 만듦 (작은 사실 번들 반환).

## 북극성 모델 (Hindsight x Letta)

혼합할 두 가지:

1. **Letta/MemGPT 스타일 제어 루프**

- 작은 "코어"를 항상 컨텍스트에 유지 (페르소나 + 핵심 사용자 사실)
- 나머지는 컨텍스트 밖에 있고 도구를 통해 검색
- 메모리 쓰기는 명시적 도구 호출 (추가/교체/삽입), 지속화 후 다음 턴에 재주입

2. **Hindsight 스타일 메모리 기판**

- 관찰된 것 vs 믿는 것 vs 요약된 것 분리
- retain/recall/reflect 지원
- 증거와 함께 진화할 수 있는 신뢰도를 가진 의견
- 엔티티 인식 검색 + 시간 쿼리 (전체 지식 그래프 없이도)

## 제안된 아키텍처 (Markdown 진실의 원천 + 파생 인덱스)

### 정식 저장소 (git 친화적)

`~/.openclaw/workspace`를 정식 사람이 읽을 수 있는 메모리로 유지합니다.

제안된 워크스페이스 레이아웃:

```
~/.openclaw/workspace/
  memory.md                    # 작음: 내구성 있는 사실 + 선호도 (코어 수준)
  memory/
    YYYY-MM-DD.md              # 일일 로그 (추가; 내러티브)
  bank/                        # "타입이 있는" 메모리 페이지 (안정적, 검토 가능)
    world.md                   # 세상에 대한 객관적 사실
    experience.md              # 에이전트가 한 것 (1인칭)
    opinions.md                # 주관적 선호/판단 + 신뢰도 + 증거 포인터
    entities/
      Peter.md
      The-Castle.md
      warelay.md
      ...
```

참고:

- **일일 로그는 일일 로그로 유지됩니다**. JSON으로 변환할 필요 없습니다.
- `bank/` 파일은 **큐레이션된** 것으로, 성찰 작업에 의해 생성되며 여전히 수동으로 편집 가능합니다.
- `memory.md`는 "작고 + 코어 수준"으로 유지: Clawd가 매 세션마다 볼 것들.

### 파생 저장소 (기계 회상)

워크스페이스 아래에 파생 인덱스 추가 (반드시 git 추적은 아님):

```
~/.openclaw/workspace/.memory/index.sqlite
```

다음으로 뒷받침:

- 사실 + 엔티티 링크 + 의견 메타데이터를 위한 SQLite 스키마
- 어휘 회상을 위한 SQLite **FTS5** (빠르고, 작고, 오프라인)
- 시맨틱 회상을 위한 선택적 임베딩 테이블 (여전히 오프라인)

인덱스는 항상 **Markdown에서 재구축 가능**합니다.

## 보존 / 회상 / 성찰 (운영 루프)

### 보존: 일일 로그를 "사실"로 정규화

여기서 중요한 Hindsight의 핵심 통찰: 작은 스니펫이 아닌 **내러티브적이고 자기 완결적인 사실**을 저장합니다.

`memory/YYYY-MM-DD.md`에 대한 실용적 규칙:

- 하루 끝에 (또는 중간에), 다음과 같은 2-5개의 글머리표를 가진 `## Retain` 섹션 추가:
  - 내러티브적 (크로스 턴 컨텍스트 보존)
  - 자기 완결적 (나중에 독립적으로 의미 있음)
  - 타입 + 엔티티 멘션으로 태그됨

예시:

```
## Retain
- W @Peter: Currently in Marrakech (Nov 27–Dec 1, 2025) for Andy's birthday.
- B @warelay: I fixed the Baileys WS crash by wrapping connection.update handlers in try/catch (see memory/2025-11-27.md).
- O(c=0.95) @Peter: Prefers concise replies (<1500 chars) on WhatsApp; long content goes into files.
```

최소 파싱:

- 타입 접두사: `W` (세상), `B` (경험/전기), `O` (의견), `S` (관찰/요약; 보통 생성됨)
- 엔티티: `@Peter`, `@warelay` 등 (슬러그가 `bank/entities/*.md`에 매핑)
- 의견 신뢰도: `O(c=0.0..1.0)` 선택적

작성자가 생각하고 싶지 않은 경우: 성찰 작업이 나머지 로그에서 이 글머리표를 추론할 수 있지만, 명시적 `## Retain` 섹션이 가장 쉬운 "품질 레버"입니다.

### 회상: 파생 인덱스에 대한 쿼리

회상이 지원해야 하는 것:

- **어휘**: "정확한 용어 / 이름 / 명령 찾기" (FTS5)
- **엔티티**: "X에 대해 알려줘" (엔티티 페이지 + 엔티티 연결 사실)
- **시간**: "11월 27일 전후에 무슨 일이 있었나" / "지난주 이후"
- **의견**: "Peter는 무엇을 선호하는가?" (신뢰도 + 증거 포함)

반환 형식은 에이전트 친화적이고 출처를 인용해야 합니다:

- `kind` (`world|experience|opinion|observation`)
- `timestamp` (원본 날짜, 또는 존재하는 경우 추출된 시간 범위)
- `entities` (`["Peter","warelay"]`)
- `content` (내러티브 사실)
- `source` (`memory/2025-11-27.md#L12` 등)

### 성찰: 안정적 페이지 생성 + 믿음 업데이트

성찰은 예약된 작업(일일 또는 하트비트 `ultrathink`)으로:

- 최근 사실에서 `bank/entities/*.md` 업데이트 (엔티티 요약)
- 강화/모순에 따라 `bank/opinions.md` 신뢰도 업데이트
- 선택적으로 `memory.md`에 대한 편집 제안 ("코어 수준" 내구성 있는 사실)

의견 진화 (간단하고 설명 가능):

- 각 의견에 포함:
  - 진술
  - 신뢰도 `c in [0,1]`
  - last_updated
  - 증거 링크 (지지 + 반박 사실 ID)
- 새 사실 도착 시:
  - 엔티티 겹침 + 유사성으로 후보 의견 찾기 (FTS 먼저, 나중에 임베딩)
  - 작은 델타로 신뢰도 업데이트; 큰 변동은 강한 모순 + 반복적 증거 필요

## CLI 통합: 독립 vs 심층 통합

권장: **OpenClaw에 심층 통합**, 하지만 분리 가능한 코어 라이브러리 유지.

### 왜 OpenClaw에 통합하는가?

- OpenClaw가 이미 알고 있는 것:
  - 워크스페이스 경로 (`agents.defaults.workspace`)
  - 세션 모델 + 하트비트
  - 로깅 + 문제 해결 패턴
- 에이전트 자체가 도구를 호출하길 원함:
  - `openclaw memory recall "..." --k 25 --since 30d`
  - `openclaw memory reflect --since 7d`

### 왜 여전히 라이브러리를 분리하는가?

- gateway/런타임 없이 메모리 로직을 테스트 가능하게 유지
- 다른 컨텍스트에서 재사용 (로컬 스크립트, 향후 데스크톱 앱 등)

형태:
메모리 도구는 작은 CLI + 라이브러리 레이어로 의도되지만, 이는 탐구적일 뿐입니다.

## "S-Collide" / SuCo: 언제 사용하는가 (리서치)

"S-Collide"가 **SuCo (Subspace Collision)**를 의미하는 경우: 이는 서브스페이스에서 학습/구조화된 충돌을 사용하여 강한 회수율/지연 트레이드오프를 목표로 하는 ANN 검색 접근 방식입니다 (논문: arXiv 2411.14754, 2024).

`~/.openclaw/workspace`에 대한 실용적 견해:

- SuCo로 **시작하지 마세요**.
- SQLite FTS + (선택적) 간단한 임베딩으로 시작; 대부분의 UX 개선을 즉시 얻을 수 있습니다.
- 다음 경우에만 SuCo/HNSW/ScaNN급 솔루션을 고려:
  - 코퍼스가 큰 경우 (수만/수십만 청크)
  - 무차별 대입 임베딩 검색이 너무 느려지는 경우
  - 회수 품질이 어휘 검색으로 의미 있게 병목이 되는 경우

오프라인 친화적 대안 (복잡도 순):

- SQLite FTS5 + 메타데이터 필터 (ML 없음)
- 임베딩 + 무차별 대입 (청크 수가 적으면 놀라울 정도로 잘 작동)
- HNSW 인덱스 (일반적, 강건; 라이브러리 바인딩 필요)
- SuCo (연구 수준; 임베드할 수 있는 견고한 구현이 있으면 매력적)

미해결 질문:

- 당신의 머신 (노트북 + 데스크톱)에서 "개인 비서 메모리"를 위한 **최고의** 오프라인 임베딩 모델은 무엇인가?
  - 이미 Ollama가 있다면: 로컬 모델로 임베드; 그렇지 않으면 도구 체인에 작은 임베딩 모델을 포함.

## 가장 작은 유용한 파일럿

최소한이지만 여전히 유용한 버전을 원한다면:

- `bank/` 엔티티 페이지와 일일 로그에 `## Retain` 섹션 추가.
- 인용(경로 + 줄 번호)이 포함된 회상을 위해 SQLite FTS 사용.
- 회수 품질이나 규모가 요구하는 경우에만 임베딩 추가.

## 참고 문헌

- Letta / MemGPT 개념: "코어 메모리 블록" + "아카이브 메모리" + 도구 주도 자기 편집 메모리.
- Hindsight 기술 보고서: "retain / recall / reflect", 4개 네트워크 메모리, 내러티브 사실 추출, 의견 신뢰도 진화.
- SuCo: arXiv 2411.14754 (2024): "Subspace Collision" 근사 최근접 이웃 검색.
