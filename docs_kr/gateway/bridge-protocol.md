# 브리지 프로토콜 (Bridge Protocol)

브리지 프로토콜은 **레거시(구형)** 노드 전송 방식(TCP JSONL)입니다. 최신 노드 클라이언트는 이 대신 통합된 [게이트웨이 WebSocket 프로토콜](/gateway/protocol)을 사용해야 합니다.

> **참고**: 현재 OpenClaw 빌드에서는 TCP 브리지 리스너를 더 이상 포함하지 않으며, 이 문서는 기술적 참조를 위해 유지됩니다. 설정 파일의 `bridge.*` 키들도 더 이상 유효하지 않습니다.

## 주요 설계 특징 (레거시)

1. **보안 경계**: 전체 게이트웨이 API 대신 미리 허가된 작은 목록(Allowlist)만 노출하여 보안을 강화했습니다.
2. **페어링 및 노드 식별**: 게이트웨이가 노드의 접근을 관리하며, 노드별 토큰과 연동하여 승인된 장치만 연결할 수 있었습니다.
3. **탐색 편의성**: Bonjour를 통해 로컬 네트워크의 게이트웨이를 자동으로 찾거나, Tailscale을 통해 직접 연결할 수 있었습니다.

## 통신 방식 (Transport)

- **형식**: TCP 연결, 한 줄당 하나의 JSON 객체 전송 (JSONL).
- **포트**: 과거 기본 포트는 `18790`이었습니다.
- **보안**: TLS 암호화를 지원했습니다.

## 핸드셰이크 및 페어링 절차

1. **Hello**: 클라이언트가 노드 정보와 토큰을 보냅니다.
2. **검증**: 페어링되지 않은 경우 게이트웨이가 에러를 반환합니다.
3. **Pair-request**: 클라이언트가 페어링 요청을 보냅니다.
4. **승인**: 사용자가 게이트웨이에서 승인하면 `hello-ok`와 함께 연결이 완료됩니다.

## 프레임 구조

- **클라이언트 → 게이트웨이**:
  - `req`/`res`: 채팅, 세션, 설정 등 제한된 RPC 호출.
  - `event`: 음성 인식 결과, 실행 상태 등 노드 신호.
- **게이트웨이 → 클라이언트**:
  - `invoke`: 캔버스(Canvas), 카메라, 위치 정보 가져오기 등 노드 제어 명령어.
  - `event`: 구독 중인 세션의 채팅 업데이트 알림.

## 현재 상태

현재 OpenClaw는 모든 노드 통신을 WebSocket 기반의 게이트웨이 프로토콜로 통합하고 있습니다. 기존 브리지 프로토콜 관련 코드는 `src/gateway/server-bridge.ts`에서 제거되었습니다.

상세한 최신 프로토콜 사양은 [게이트웨이 프로토콜](/gateway/protocol) 문서를 참고하십시오.
