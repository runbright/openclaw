# 게이트웨이 트러블슈팅 (Troubleshooting)

이 문서는 OpenClaw 운영 중 발생할 수 있는 주요 문제들에 대한 심층 해결 가이드입니다. 더 가벼운 진단 단계가 필요하다면 [빠른 도움말](/help/troubleshooting)을 먼저 확인하십시오.

## 진단 기본 명령어

문제가 생기면 먼저 다음 명령어들을 순서대로 실행해 보세요:
```bash
openclaw status               # 전체 요약
openclaw gateway status       # 게이트웨이 프로세스 상태
openclaw logs --follow        # 실시간 로그 감시
openclaw doctor               # 시스템 종합 진단 및 자동 복구
openclaw channels status --probe # 채널 연결 상태 상세 확인
```

## 주요 증상별 대처법

### 1. 봇이 대답을 하지 않을 때
채널은 연결된 것 같은데 응답이 없다면 라우팅 정책이나 허용 목록 설정을 확인해야 합니다.
- **체크**: `openclaw pairing list <채널>` 명령어로 보낸 사람이 승인되었는지 확인하세요.
- **체크**: 그룹 채팅의 경우 멘션 규칙(`requireMention`)에 맞게 보냈는지 로그를 확인하세요.

### 2. 게이트웨이 서비스 실행 실패
게이트웨이가 시작되지 않거나 자꾸 죽는 경우입니다.
- **`EADDRINUSE`**: 이미 다른 프로그램이 해당 포트(기본 18789)를 쓰고 있습니다.
- **`refusing to bind ... without auth`**: 외부 접속을 허용하면서 인증 토큰을 설정하지 않은 경우 보안상 실행이 거부됩니다.
- **해결**: `openclaw doctor`를 실행하여 제안하는 수리 단계를 따르십시오.

### 3. 하트비트나 크론 잡이 작동하지 않을 때
예약된 작업이나 하트비트 알림이 오지 않는 경우입니다.
- **체크**: `openclaw cron status`로 스케줄러가 활성화되어 있는지 보세요.
- **체크**: `activeHours` 설정 때문에 현재 시간이 활동 금지 시간대인지 확인하세요.

### 4. 원격 접속 UI 연결 안 됨
대시보드나 제어 UI가 게이트웨이에 연결되지 않는 경우입니다.
- **체크**: 주소(IP)와 포트가 맞는지, 인증 토큰이 일치하는지 확인하세요.
- **체크**: HTTPS 환경이 아닌 곳에서 기기 인증 정보가 누락되었을 수 있습니다.

### 5. 브라우저 도구 오류
브라우저 제어 기능이 작동하지 않는 경우입니다.
- **체크**: `openclaw browser status`로 브라우저 실행 경로가 맞는지 확인하세요.
- **체크**: 리눅스 환경이라면 필요한 라이브러리가 누락되었을 수 있습니다.

## 업그레이드 후 갑자기 안 될 때
대부분의 원인은 이전 버전과 달라진 설정값이나 강화된 보안 정책 때문입니다.
- `openclaw gateway install --force` 명령어로 시스템 서비스를 다시 설치해 보세요.
- `openclaw doctor`가 알려주는 마이그레이션 단계를 수행하세요.

상세한 에러 로그 시그니처나 파일 기반 디버깅 방법은 [영문 원본 문서](/gateway/troubleshooting)를 참고하십시오.
