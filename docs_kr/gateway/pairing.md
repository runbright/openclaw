# 게이트웨이 기반 페어링 (Gateway-Owned Pairing)

게이트웨이 기반 페어링 방식에서는 **게이트웨이**가 어떤 노드의 접속을 허용할지에 대한 최종 결정권을 가집니다. macOS 앱이나 다른UI 클라이언트는 보류 중인 요청을 단순히 승인하거나 거절하는 역할만 수행합니다.

## 핵심 개념

- **보류 중인 요청 (Pending request)**: 노드가 게이트웨이에 가입을 요청한 상태이며, 관리자의 승인이 필요합니다.
- **페어링된 노드 (Paired node)**: 승인이 완료되어 인증 토큰을 발급받은 노드입니다.
- **보안**: 페어링 요청은 **5분**이 지나면 자동으로 만료됩니다.

## 페어링 절차

1. **요청**: 노드(iOS 앱 등)가 게이트웨이 WebSocket에 접속하여 페어링을 요청합니다.
2. **알림**: 게이트웨이는 요청을 저장하고 `node.pair.requested` 이벤트를 발생시킵니다.
3. **결정**: 관리자가 CLI나 UI를 통해 해당 요청을 승인하거나 거절합니다.
4. **발급**: 승인 시 게이트웨이는 **새로운 토큰**을 생성하여 노드에 전달합니다. (토큰은 페어링할 때마다 갱신됩니다.)
5. **연결**: 노드는 발급받은 토큰을 사용하여 다시 접속하고 "페어링됨" 상태가 됩니다.

## CLI 명령어 (서버 환경 권장)

```bash
# 보류 중인 페어링 요청 목록 확인
openclaw nodes pending

# 특정 요청 승인
openclaw nodes approve <요청ID>

# 특정 요청 거절
openclaw nodes reject <요청ID>

# 페어링된 노드 목록 및 상태 확인
openclaw nodes status

# 노드 이름 변경 (예: "거실 아이패드")
openclaw nodes rename --node <아이디> --name "거실 아이패드"
```

## 데이터 저장소

페어링 데이터는 게이트웨이 상태 디렉토리(`~/.openclaw`) 아래에 저장됩니다:
- `~/.openclaw/nodes/paired.json`: 승인된 노드 및 토큰 정보 (민감 정보 포함)
- `~/.openclaw/nodes/pending.json`: 승인 대기 중인 요청 정보

> **보안 주의**: `paired.json` 파일에는 노드 인증 토큰이 들어있으므로 외부로 유출되지 않도록 주의하십시오. 토큰을 갱신하려면 노드 항목을 삭제하고 다시 페어링해야 합니다.

상세한 API 명세나 자동 승인 조건은 [영문 원본 문서](/gateway/pairing)를 참고하십시오.
