# 보안 (Security) 🔒

OpenClaw는 명령줄 접근 권한을 가진 강력한 AI 모델을 실제 메시징 서비스와 도구에 연결하는 도구입니다. 따라서 "완벽한 보안"은 존재하지 않으며, 다음 세 가지 핵심 질문을 바탕으로 신중하게 설정해야 합니다:
1. **누가** 내 봇과 대화할 수 있는가?
2. **어디서** 봇이 행동할 수 있는가?
3. 봇이 **무엇을** 만질 수 있는가?

## 보안 진단: `openclaw security audit`

가장 먼저 기억해야 할 명령어입니다. 설정 변경 후나 네트워크 노출 전후에 수시로 실행하세요.
```bash
openclaw security audit --deep
```
이 명령어는 게이트웨이 인증 누락, 브라우저 제어 노출, 파일 권한 오류 등 흔히 발생하는 보안 취약점을 자동으로 점검합니다.

## 핵심 배포 원칙

1. **개인 보조 도구 모델**: OpenClaw는 다중 사용자용 플랫폼이 아닌 "개인 비서" 모델로 설계되었습니다.
2. **단일 호스트, 단일 사용자**: 하나의 시스템(또는 VM/VPS)에는 한 명의 사용자만 접속하는 것이 권장됩니다. 여러 명이 함께 쓰는 공유 환경은 권장되지 않습니다.
3. **신뢰 경계**: 게이트웨이의 설정 파일(`openclaw.json`)이나 데이터 폴더(`~/.openclaw`)를 수정할 수 있는 사람은 모든 권한을 가진 관리자와 다름없습니다.

## 위험 요소: 공유 메시징 환경 (Slack, 가 가계 등)

"누구나 봇에게 말을 걸 수 있는" 환경은 다음과 같은 위험이 있습니다:
- 허용된 사용자는 누구나 봇을 통해 쉘 명령어 실행, 브라우저 조작, 파일 읽기/쓰기 도구를 작동시킬 수 있습니다.
- 한 명의 악의적인 사용자가 프롬프트 인젝션을 통해 공유 데이터나 기기를 장악할 수 있습니다.
- 개인적인 목적의 에이전트와 공용 업무용 에이전트가 같은 게이트웨이에서 돌아간다면 데이터 유출 위험이 커집니다.

**해결책**: 개인용 데이터 에이전트는 철저히 비공개로 유지하고, 팀 공용 에이전트는 최소한의 도구만 허용하여 별도의 게이트웨이나 VPS에서 격리 운영하세요.

## 60초 만에 끝내는 보안 강화 설정

다음 설정을 `openclaw.json`에 적용하면 가장 보수적이고 안전한 상태로 시작할 수 있습니다:

```json5
{
  "gateway": {
    "mode": "local",
    "bind": "loopback", // 로컬 접속만 허용
    "auth": { "mode": "token", "token": "길고_복잡한_랜덤_토큰" }
  },
  "session": {
    "dmScope": "per-channel-peer" // 사용자별 세션 분리
  },
  "tools": {
    "profile": "messaging", // 메시지 전송 기능만 활성화
    "deny": ["group:automation", "group:runtime", "group:fs"], // 위험 도구 차단
    "fs": { "workspaceOnly": true }, // 파일 접근을 워크스페이스로 대기
    "exec": { "security": "deny", "ask": "always" } // 명령어 실행 시 매번 확인
  }
}
```

## 주요 보안 기능 가이드

### 1. 샌드박싱 (Sandboxing)
모델이 사용하는 도구를 도커(Docker) 컨테이너 안에서 격리 실행하여 시스템 전체가 오염되는 것을 방지합니다. [샌드박싱 문서](/gateway/sandboxing)를 참고하여 활성화하십시오.

### 2. 프롬프트 인젝션 방어
프롬프트 인젝션은 완벽히 막을 수 있는 기술이 아닙니다. 따라서 다음 원칙을 따르세요:
- 봇이 외부 웹사이트나 문서를 읽어야 한다면, 파일 쓰기/실행 권한이 없는 **읽기 전용 에이전트**에게 먼저 요약을 시키세요.
- 최신 고성능 모델(Anthropic Claude 3.5 Sonnet 등)일수록 인젝션 저항력이 강합니다.
- 중요한 비밀 정보(API 키 등)는 에이전트 프롬프트에 직접 적지 말고, 게이트웨이 설정 파일이나 환경 변수로 관리하세요.

### 3. 브라우저 제어 보안
브라우저를 통해 웹 서핑을 할 수 있게 허용하면 에이전트가 여러분의 계정으로 로그인된 웹 서비스에 접근할 수 있습니다.
- **전용 프로필 사용**: 여러분이 평소 쓰는 브라우저 프로필 대신, OpenClaw 전용의 빈 프로필을 사용하세요.
- **SSRF 정책**: 내부 시스템(localhost 등)으로의 접근을 차단하려면 `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork`를 `false`로 설정하십시오.

### 4. 네트워크 노출 주의
- **Tailscale 활용**: 포트 포워딩으로 공용 인터넷에 게이트웨이를 여는 대신, Tailscale Serve를 사용하여 안전한 내부망 주소로만 노출하세요.
- **SSH 터널링**: 원격에서 직접 접속할 때는 SSH 터널을 통하는 것이 가장 표준적인 방법입니다.

## 저장소 관리 (민감 데이터 위치)

다음 데이터에는 비밀 정보가 포함되어 있으므로 백업이나 공유 시 각별히 주의하세요:
- `~/.openclaw/openclaw.json`: 설정 및 토큰
- `~/.openclaw/credentials/`: 채널 인증 정보 (WhatsApp 기기 정보 등)
- `~/.openclaw/agents/<아이디>/sessions/`: 대화 기록 (평문 저장)

## 사고 대응 절차

봇이 의도치 않은 작업을 수행했거나 공격을 받은 것으로 의심된다면:
1. **즉시 중단**: 게이트웨이 프로세스를 즉시 종료하십시오.
2. **네트워크 차단**: `bind: "loopback"`으로 설정을 변경하여 외부 접근을 막으세요.
3. **인증 정보 갱신**: 게이트웨이 토큰, 채널 API 키 등 모든 인증 수단을 새로 발급받아 교체하십시오.
4. **로그 분석**: `/tmp/openclaw/` 폴더 내의 로그를 분석하여 피해 범위를 확인하십시오.

상세한 보안 감사 항목 설명이나 기술 명세는 [영문 원본 문서](/gateway/security)를 참고하십시오.
