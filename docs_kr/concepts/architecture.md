# 게이트웨이 아키텍처 (Gateway Architecture)

OpenClaw의 핵심 시스템 구조와 작동 방식에 대한 설명입니다.

## 개요

- **게이트웨이(Gateway)**: 모든 메시징 플랫폼(WhatsApp, Telegram, Slack, Discord 등)과의 연결을 독점적으로 관리하는 단일 수신 대기 프로세스입니다.
- **WebSocket 통신**: macOS 앱, CLI, 웹 UI, 자동화 도구 등의 클라이언트는 설정된 주소(기본값 `127.0.0.1:18789`)의 WebSocket을 통해 게이트웨이에 접속합니다.
- **노드(Nodes)**: macOS, iOS, Android 기기들이 '노드' 역할로 게이트웨이에 연결되어 카메라, 화면 기록, 위치 정보 등의 기능을 제공합니다.
- **캔버스 호스트**: 게이트웨이의 HTTP 서버가 에이전트가 수정 가능한 HTML/CSS/JS 콘텐츠(`/__openclaw__/canvas/`)를 서빙합니다.

## 시스템 구성 요소

### 1. 게이트웨이 데몬 (Gateway Daemon)
- 외부 서비스(메신저)와의 연결을 유지합니다.
- 상의이 정의된 WebSocket API를 통해 요청을 처리하고 이벤트를 푸시합니다.
- 수신되는 데이터가 정해진 규격(JSON Schema)에 맞는지 검증합니다.

### 2. 클라이언트 (macOS 앱 / CLI / 웹 관리자)
- 게이트웨이와 1:1 WebSocket 연결을 맺습니다.
- 건강 상태 확인, 메시지 전송, 에이전트 실행 등의 요청을 보냅니다.
- 게이트웨이가 보내는 실시간 이벤트(대화 내용, 상태 변화 등)를 구독합니다.

### 3. 노드 (Nodes)
- 게이트웨이에 `role: node`로 연결되는 장치들입니다.
- 기기 기반의 페어링 절차를 통해 신뢰 관계를 형성합니다.
- 기기의 하드웨어적 기능(카메라 활용, 화면 캡처 등)을 에이전트가 사용할 수 있게 합니다.

## 연결 생명주기 (Connection Lifecycle)

1. **연결 요청 (`connect`)**: 클라이언트가 게이트웨이에 접속을 시도합니다. 첫 프레임은 반드시 `connect`여야 합니다.
2. **핸드쉐이크**: 토큰 인증 및 보안 챌린지를 거쳐 연결이 승인됩니다. 승인되면 현재 시스템의 상태(건강도, 접속자 정보 등) 스냅샷을 받습니다.
3. **이벤트 구독**: 이후 발생하는 대화 내용, 에이전트의 작업 진행 상황 등을 실시간으로 수신합니다.

## 보안 및 신뢰 모델

- **장치 토큰**: 모든 WebSocket 클라이언트는 장치 식별자를 포함해야 하며, 처음 연결 시 페어링 승인이 필요합니다.
- **로컬 신뢰**: 게이트웨이가 실행 중인 컴퓨터 내부(loopback)에서의 연결은 편의를 위해 자동 승인될 수 있습니다.
- **인증 토큰**: 외부 원격 접속 시에는 반드시 비밀 토큰(`OPENCLAW_GATEWAY_TOKEN`)이 일치해야 합니다.
