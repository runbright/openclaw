# 명령 대기열 (Command Queue)

에이전트가 답변을 작성하는 동안 추가 메시지가 도착하면, OpenClaw는 이를 **명령 대기열**에 넣고 순차적으로 처리합니다. 이는 여러 작업이 서로 충돌하지 않게 하고, 시스템 자원(세션 파일, 로그 등)을 안전하게 관리하기 위함입니다.

## 작동 방식

- **세션당 하나의 작업**: 동일한 대화 세션에서는 한 번에 하나의 에이전트 실행 작업만 허용됩니다.
- **글로벌 동시성 제어**: 전체 시스템에서 동시에 실행될 수 있는 에이전트 작업의 총 개수를 제어합니다. (`agents.defaults.maxConcurrent`로 설정 가능)
- **실시간 반응**: 대기열에서 순서를 기다리는 동안에도 채널이 지원한다면 "입력 중..." 표시를 즉시 띄워 사용자 경험을 유지합니다.

## 대기열 처리 모드 (Queue Modes)

에이전트가 바쁠 때 들어온 메시지를 어떻게 처리할지 선택할 수 있습니다:

- **`collect` (기본값)**: 현재 답변이 끝날 때까지 기다렸다가, 대기 중인 모든 메시지를 **하나로 묶어서** 한 번에 처리합니다. (가장 효율적)
- **`steer`**: 답변을 작성하는 도중에 새 메시지 내용을 주입하여 답변 방향을 실시간으로 수정(조정)합니다.
- **`followup`**: 현재 답변이 끝나면 바로 이어서 다음 메시지에 대한 답변을 시작합니다.
- **`interrupt`**: 현재 진행 중인 작업을 즉시 중단하고 가장 마지막에 온 메시지를 새로 처리합니다.

## 유용한 설정 및 명령어

### 세션별 설정 변경
대화 도중에 명령어를 입력하여 대기열 방식을 바꿀 수 있습니다:
- `/queue collect`: 현재 세션의 대기열 모드를 '수집 후 처리'로 설정합니다.
- `/queue steer`: 현재 세션의 대기열 모드를 '실시간 조정'으로 설정합니다.

### 상세 옵션
- **`debounceMs`**: 추가 메시지가 더 오는지 기다리는 짧은 시간입니다. (기본 1초)
- **`cap`**: 대기열에 쌓을 수 있는 최대 메시지 개수입니다. (기본 20개)
- **`drop`**: 대기열이 가득 찼을 때 처리 방식입니다. (예: `summarize` - 넘친 메시지들을 요약해서 나중에 한꺼번에 전달)

## 문제 해결

- 답변이 너무 늦게 온다면 `openclaw logs --follow` 명령어로 "queued for ...ms"라는 로그가 찍히는지 확인하세요. 대기열이 밀리고 있다는 신호일 수 있습니다.
- 동시 처리를 늘리려면 설정 파일에서 `maxConcurrent` 값을 높여보세요.
